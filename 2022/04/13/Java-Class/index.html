1. 제어자

클래스, 필드, 메서드, 생성자 등에게 특징을 부여하는 문법

1.1 Access modifier

  • private < package(생략가능) < protected < public
    • private: 자기 클래스내에서만 접근 가능
    • package : 같은 패키지 안에서만 접근 가능
    • protected : 다른 패키지일지라도 상속관계가 있으면 접근 가능
    • public : 어디서나 접근 가능
  • 대상 = {필드, 메서드, 생성자} + 클래스
    • 클래스의 경우 package와 public을 구분해서 쓴다.
    • 다른 패키지에서 임포트를 허용할 클래스라면, public을 써야만한다.

1.2 others

  • static : 멤버를 객체가 아닌 클래스에 종속되게 함
    • 대상 = {필드, 메서드}
  • final : 최종
    • 대상 = {필드, 메서드, 지역변수, 클래스}
      • 변수의 경우 한번 대입된 값을 수정하지 못함 (상수화)
        • 지역변수 final의 경우, 불변성을 위해 씀
        • final
      • 메서드의 경우 오버라이딩 금지
      • 클래스의 경우 상속 금지
  • abstract
    • 대상 = {클래스, 메서드}

2. Class

  • 클래스의 구성

    • 클래스 헤더

    • 멤버 필드

      • 인스턴스 멤버 필드

      • 스태틱 멤버 필드

        • 상수

          public static final

        • 싱글톤 디자인

    • 스태틱 블록 : 클래스 로드 시점에 실행되는 블록, 주로 정적필드 초기화용으로 씀

    • 생성자

    • 멤버 메서드

      • 인스턴스 메서드 : 스태틱 멤버와 인스턴스 멤버에 접근이 가능함
      • 스태틱 메서드 : 스태틱 멤버만 접근 가능

    • 이너클래스

2.1 필드 vs 지역변수

  • 필드
    • 클래스에 포함된 변수
    • 인스턴스 변수는 힙영역에 객체의 내부에 저장된다
    • 자동 초기화
  • 지역변수
    • 메서드에 포함된 변수
    • 지역변수는 스택영역에 저장되고 unreachable 상태가 되면 자동으로 삭제됨

2.2 오버로딩, 오버라이딩

  • 오버로딩 : 같은 메서드명, 다른 동작

    • 메서드 시그니처가 identifier가 된다.
      • 메서드명, 인자수, 인자타입

  • 오버라이딩 : 상속받은 메서드를 재정의

2.3 상속

  • 상속하는 이유

    • 코드 중복성 제거

    • 다형성(polymorphism)

      • 농장에는 딸기, 바나나, 귤 등 다양한 종류의 과일이 있다.

      • 다양한 종류의 객체들을과일 배열에 담을 수 있다.

      • Fruit strawberry1 = new Strawberry();

        • 이는 유효한 다형적 표현이다. (업캐스팅)
        • Strawberry에서 추가한 멤버는 접근이 안된다.
        • 대신, Strawberry에서 오버라이딩한 메소드는 접근이 된다.
          • 이게 추상클래스, 인터페이스에서 선언만 하는 근본적 이유
        • 업캐스팅하며 생성된 객체라면 원래 상태로 다운캐스팅 가능

  • 메모리 구조

    • 자식의 생성자에선
      • 디폴트로 super()를 먼저 호출한 후에
      • 생성자 내부가 실행된다.
    • 자식 객체 내부에, 부모 객체가 있는 구조이다.

  • 오버라이딩

    • 부모의 접근 지정자에 따른 자식 접근 지정자
      • 자식은 접근 자격을 너그럽게 해줄 순 있어도 엄격하게 할 순 없다.
        • 부모가 public이면 자식도 public이여야함
        • 부모가 private면 자식은 아무거나 해도 됨

2.4 추상클래스 & 인터페이스

2.4.1 추상클래스

  • 불완전한 설계도
    • 개발자에게 명시하는 기능을 한다.
  • 추상메소드를 포함한 클래스이다.
    • 추상메소드 없으면 굳이 abstract 쓸 이유 없음
    • abstract int func(int a, int b);
    • 상속받은 클래스에게 오버라이딩을 해야한다는 의미가 전달됨 (구현 안하고 추상메소드 호출 시 에러)
  • 특징
    • 인스턴스화 불가능, 대신 업캐스팅은 가능
    • 내부에 추상 메소드를 구현 가능

2.4.2 인터페이스

  • 내부의 모든 필드가 public static final로 정의된다.

    public static final 생략해도 컴파일러가 붙여주긴 하지만, 그냥 의존하지 말고 직접 쓰길 바람

  • 모든 메서드는 public abstract로 정의된다.

    자바 8부턴 디폴트, 정적 메소드 추가

  • 다중 상속이 허용된다.

    • 구현부가 없으니 메소드 충돌날 일이 없다

  • 추상메소드를 구현안하면 문법오류가 발생한다.

    당연히 자식에서 오버라이딩할 때 메소드의 Access modifier는 public일 수 밖에 없다.

  • 디폴트메소드의 동기

    • 인터페이스를 상속한 클래스가 100개 있다고 해보자.
    • 업데이트를 하면서 100개의 클래스가 공유하는 메소드를 추가하고 싶다고 생각하자.
    • 그러면 2가지 선택지가 있다.
      • 인터페이스를 추상클래스로 바꾸고, … (100개의 클래스가 이미 다른 추상클래스들을 상속하고 있다면 ㅅㄱ)
      • 100개의 클래스에 메소드를 복붙한다. (중복)
    • 이러한 문제를 해결하려면 인터페이스에 디폴트메소드를 추가하자.
      • 인터페이스명.super.디폴트메소드명()

  • 정적메소드

    • 인터페이스명.디폴트메소드명()

3. 제네릭

객체를 언급할 떄, 타입 = 클래스 는 똑같은 말임

다양한 종류의 객체를 안정적으로(Type conversion 없이), 중복없이(오버로딩 없이) 다룰 수 있게 해준다.

  • 동기
    • 다양한 인터페이스와 추상 클래스가 있다.
    • 그로 인해, 자바엔 다양한 클래스가 존재하게 된다.
    • 생성자, 메서드 등 오버로딩 해야할 게 너무 많아진다

3.1 제네릭 클래스

클래스 내에서 <제네릭타입>는 이미 존재하는 클래스처럼 쓰인다.

물론 객체 생성 시점에서 타입클래스가 정해진다.

제네릭 클래스를 활용하면 다양한 객체를 저장하고 관리하는 컨테이너를 만들 때 유용하다.

  • T : 타입
  • K : 키
  • V : 값
  • N : 숫자
  • E : 원소

static 멤버는 타입파라미터 쓸 수 없다.

static 제네릭 메서드는 만들 수 있긴 하다.

3.3 타입 제한

타입은 객체 생성 시점에서 정해진다.

그러므로, Object에 없는 멤버를 참조하면 컴파일 에러가 발생한다.

또한, 컨테이너로서 과일에 관련한 객체만 관리하고 싶을 수 있다.

동물은 못넣게 하려면 제네릭타입에 제한을 줘야한다.

  • 한정된 타입 파라미터(Bounded Type Parameter)
    • <T extends 최상위 타입>
    • <T super 최하위 타입>
  • 와일드카드 타입 : 모든 타입 s.t.
    • <?>
    • <? extends 최상위 타입>
    • <? super 최하위 타입>

<?> 와 Object는 다르다

Integer는 Object이지만, List는 Obejct가 아님
