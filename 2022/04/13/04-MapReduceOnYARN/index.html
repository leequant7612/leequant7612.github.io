3. 맵리듀스 작동방식

3.1 MapReduce on YARN

용어 정리

  • 입력스플릿 : 맵퍼 객체하나가 처리하게 될 블록 단위 데이터
  • 컨테이너 : 시스템자원 / NM은 컨테이너 단위로 APP을 실행한다

맵태스크는 입력스플릿이 로컬디스크에 있을 때 최적이다.

그렇기에 스플릿크기 최대크기를 블록크기에 맞춰져있는게 디폴트이다.

모든 잡태스크는 상태정보를 갖는다.

  • 상태 : 성공, 실패, 진행중
  • 진행상황 : 태스크 완료 비율, …
  • 카운터 값
  • 상태메세지

이들은 리소스 매니저 웹 UI에서 확인할 수 있다.

3.2 Shuffle on MapReduce

셔플 : 맵퍼의 출력을 키값 기준으로 병합시켜서 리듀서의 입력으로 넣는 과정

용어정리

  • 파티션 = 리듀서의 인풋
  • 파티션 수 = 리듀서 수
  • 병합계수 = 맵아웃풋 몇개 모으고 병합할 지 / mr.task.io.sort.factor / 디폴트 10

가능하면 셔플링(Sort phase)을 메모리상에서만 하고 싶으니 셔플에 많은 메모리 할당 추천

맵리듀스 함수를 작성할 땐 적은 메모리를 사용하게끔 작성하는 것 역시 중요

4.1 기본 맵리듀스 잡

4.2 입력포맷

하둡은 일반텍스트파일에서 데이터베이스에 이르기까지 다양한 유형의 데이터 포맷을 입력받을 수 있다.

4.2.1 입력스플릿 ==> 레코드

  • 입력 스플릿(입력 청크)은 여러개의 레코드로 나눠질 수 있으며,
  • Mapper는 하나의 레코드에서 키-값쌍을 뽑아내고 처리한다.

레코드가 Mapper에 오기 전에 **InputFormat**은

  • context와 상호작용하여
  • 입력스플릿을 생성하고
  • 레코드를 분리하는 역할을 한다.

과정을 좀 더 디테일하게 적으면,

  1. ClientApp에서 getSplits()을 호출하면 MRAppMaster은 인풋스플릿들을 얻는다.

  2. 인풋스플릿들에 기반해 맵태스크를 스케줄링한다.

  3. 맵태스크는 InputFormat.createRecordReader()를 호출해 Record Iterator를 얻는다.

  4. 맵태스크는 Record Iterator를 사용해서 레코드의 키 값 쌍들을 생성한다.

Map 연산에는 오버헤드가 거의 없기 때문에 MultithreadedMapper는 거의 쓰지 않는다.

다만, 맵 연산 중간에 외부서버와의 통신이 있다면 고려할만하다.

4.2.2 FileInputFormat

FileInputFormat<K, V>은 파일을 원본데이터로 사용하는 모든 InputFormat 구현체의 기본클래스다.

  • FileInputFormat<K, V>
    • TextInputFormat
    • KeyValueTextInputFormat
    • NLineInputFormat
    • CombineFileInputFormat<K, V>
    • SequenceFileInputFormat<K, V>
      • SequenceFileAsBinaryInputFormat
      • SequenceFileAsTextInputFormat
      • SequenceFileAsTextInputFilter<K, V>

FileInputFormat : 입력 경로

사용자는 드라이버 Application을 통해 입력경로의 집합을 입력한다.

유연한 제어를 위해 FileInputFormat은 4가지 정적 메소드를 제공한다.

FileInputFormat : 입력 스플릿

대부분의 Application에서 스플릿의 크기는 HDFS 블록 크기와 똑같이 둔다.

왜냐면, 맵 태스크와 동일한 로컬에 존재하는 블록 수가 줄어들기 떄문이다.

CombineFileInputFormat<K, V>

하둡은 작은 파일이 많을 때보단, 큰 파일이 적을 때 더 좋은 성능을 보인다.

왜냐면, 빈공간의 스플릿(블록)이 많다면 맵태스크가 많아지기에 오버헤드가 발생한다.

FileInputFormat은 작은파일 하나에 하나의 스플릿을 생성한다.

반면에, CombineFileInputFormat<K, V>는 여러개의 작은파일을 하나로 묶어서 스플릿을 생성한다.

블록의 노드(+랙) 지역성을 고려하여 하나의 스플릿으로 묶는다.

그러나 이 방법보단, 하나의 순차파일로 병합하는 방식을 추천한다.

이미 HDFS에 많은 작은 파일로 올라와있다면, CombineFileInputFormat를 쓰는 것도 나쁘진 않다.

스플릿 방지 : 맵 연산에 파일하나를 입력받고 싶다면

일부 Application에선 스플릿을 하지 않고, 단일 매퍼가 입력파일 전체를 처리하길 원한다.

예를 들어, 파일의 모든 레코드가 정렬되어있는 지 확인하는 방법은 이전 레코드와 값비교를 순차적으로 하는 것이다.

2가지 방법이 있다.

  1. 최소 스플릿 크기(mapreduce.input.fileinputformat.split.minsize)를 Long.MAX_VALUE로 설정

  2. 혹은 구체화된 서브클래스에 isSplitable() 메소드를 false 반환하도록 오버라이딩 하는 것이다.

파일 전체 ==> 하나의 레코드

파일 전체내용을 하나의 레코드로 받고 싶을 때는 FileInputFormat를 상속받고

  • 한번에 파일 컨텐츠를 전달하는 RecordReader를 반환하도록 createRecordReader를 오버라이딩하면 된다.

그리고, 파일 컨텐츠를 전체를 전달하는 RecordReader를 작성하자.

  • nextKeyValue를 잘 오버라이딩하면 된다.

입력스플릿 파일 정보 얻기

Mapper와 상호작용하는 context 객체의 getInputSplit() 메서드를 호출해 스플릿에 대한 정보를 얻을 수 있다.

또한, 그렇게 받은 인풋스플릿에서 FileSplit으로 형변환할 경우, 파일 정보에 접근할 수 있다.

  • getPath() : 처리할 입력파일의 경로
  • getStart() : 파일의 시작으로 부터 스플릿의 시작지점까지의 바이트 오프셋
  • getLength() : 스플릿의 바이트길이

4.2.3 텍스트 입력

하둡은 비정형 텍스트를 처리하는데 탁월하다.

TextInputFormat

  • 하둡의 기본 InputFormat이다.
  • 각 레코드는 입력 데이터의 한 행이다.
  • 키(LongWritable)는 행의 시작점까지의 바이트 오프셋이다.
  • 값(Text)은 행 내용이다.

행의 끝이 스플릿의 끝이 아닌 경우는 빈번하게 있으나, offset을 계속 끌고가기 때문에 추가로 이어서 읽는 것은 약간의 오버헤드만 가질 뿐 큰 문제는 없다.

행의 길이가 너무 길어서 메모리에 영향을 줄 수 있는 경우엔, mapreduce.input.linerecordreader.line.maxlength를 설정하자.

KeyValueTextInputFormat

위와 같은 탭으로 구분지어지는 separator가 있다면,

separator를 설정하고 싶다면, mapreduce.input.keyvaluelinerecordreader.key.value.separator를 변경한다.

NLineInputFormat

매퍼가 받게될 스플릿의 행의 개수를 지정할 수도 있다.

N값이 2라면 2번에 걸쳐서 받게 된다.

소량의 데이터를 받아서 오랜 연산을 수행하는 애플리케이션에서 사용할 수 있다.

  • OutputFormat<K, V>

4.3.1 텍스트 출력 TextOutputFormat

  • 하둡의 기본 출력 포맷
  • 어떤 타입의 키와 값을 쓰더라도 변환 가능 (내부적으로 toString() 로 변환)
  • 레코드 출력시 탭으로 키값쌍을 구분짓기 때문에, 다시 읽을 땐 KeyValueTextInputFormat

4.3.2 바이너리 출력 SequenceFileOutputFormat

  • 시퀀스 파일에 출력 결과 기록
  • 압축 용이

4.3.3 다중 출력 FileOutputFormat

  • 출력 디렉터리에 다수의 파일을 생성한다.
  • 리듀서 하나당 하나의 파일

예제 : 데이터 파티셔닝

기상데이터셋을 같은 관측소별로 파티셔닝을 하기 위해선 두개의 작업이 필요하다.

  1. 관측소 별로 파티션이 나뉘도록 파티셔너를 새로 만들고 잡의 파티셔너로 사용한다.

  2. 리듀서의 숫자를 관측소의 숫자만큼 만들고 FileOutputFormat을 사용해 파일을 출력한다.

이 방식에는 2가지 문제가 있다.

  • 전체 관측소 숫자가 몇개인 지 모르며, 메타데이터에 적혀있더라도 일치한다는 보장이 없다.
  • 불균등한 파티션 문제로 인해, 적게 일하는 리듀서수가 많다면 태스크 실행 오버헤드가 생긴다.

왠만하면, 파티션의 수는 클러스터에게 맞기는 것이 훨씬 계산적으로 효율적이다.

파티션 내에서 리듀서가 다수의 파일을 기록하는 것이 필요하다.

4.3.4 다중 출력 MultipleOutputs

  • 리듀서, 또는 맵단독잡의 매퍼가 하나 이상의 파일을 생성하게 해준다.
  • name-r-nnnnn / name-m-nnnnn
