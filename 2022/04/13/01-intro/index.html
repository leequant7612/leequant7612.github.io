2. Why parallel processing?

일반적인 분석 사이클

  • 데이터를 저장하고
  • 스토리지에서 컴퓨팅 용량(메모리 포함)으로 데이터를 이동한 다음
  • 중요한 결과가 계산되면 다시 스토리지로 이동

많은 계산량을 요구하는 빅데이터 처리에는 쪼갠 후에 동일한 성능으로 처리하는 것이 효율적이다.

  • 처리시간이 줄어든다.
  • 개별 execution node에서 실행되기 때문에 메모리 및 요구 사항이 낮다.
  • Processing network에 넣고 빼고가 자유로움으로 인한 비용절감

Single node를 스케일링 업을 하는 전략도 있다.

더 나은 전략은 클러스터 구축을 하는 것이다. horizontal scaling

이는 embarrassingly parallel 문제를 푸는 것에 효율적이다.

EP 문제는 독립적인 업무를 수행하는 경우를 의미한다.

예를 들어, 서로다른 node에 저장된 빅데이터 chunk들의 특정 칼럼에 있는 형식을 바꾸는 문제가 있다.

더 복잡한 경우엔, 다른 노드의 상태를 인식할 필요가 있다.

모든 노드가 접근 가능한 파일시스템에 읽고 쓰는 것을 요구하고 이는 복잡도가 증가한다.

Hadoop 에코시스템에서 “bringing compute to the data”은 클러스터 설계에 대한 핵심 개념이다.

  • 데이터의 특정 조각 또는 파티션에 대한 계산이 가능한 경우
  • 데이터의 해당 위치에서 계산되도록 설계되었습니다.

하둡 HDFS는 parallel computing에서 Fault tolerance도 큰 강점이다.

P1, P2, P3 파티션을 갖고있는 노드가 고장나면

새로운 노드를 추가하면 클러스터를 바로 복구할 수 있다.

3. What is Hadoop?

서로 연결된 분산 파일 시스템에서 대용량 데이터를 처리하는 데 사용된다.

서버는 클러스터들에서 application을 실행할 수 있습니다.

하둡에서 클러스터는 동시에 같이 일하는 컴퓨터들의 모음이다.

하둡은 데이터베이스가 아니라 프로세스와 작업을 병렬 또는 동시에 처리할 수 있는 생태계라는 점에 유의해야 한다.

하둡은 3가지의 핵심 구성요소로 이뤄져있다.

  • HDFS
    • HDFS는 일반 하드웨어에서 large dataset을 다루게 한다.
    • 단일 클러스터에서부터 수천개의 클러스터를 사용할 수 있다.
  • MapReduce
    • 맵리듀스는 대량의 데이터를 더 작은 단위로 분할해 동시에 처리한다.
  • YARN
    • HDFS에 저장된 데이터를 batch processing, stream processing, interactive processing, and graph processing 하기 위해 RAM과 CPU를 준비한다.

하둡엔 몇가지 결함이 존재한다.

  • Transaction (정보 교환)에 좋지 않다.
  • 병렬로 처리될 수 없는 경우 좋지 않다.
  • Low latency, 실시간 서빙에 적합하지 않다.
  • 다양한 작은 파일에 적합하지 않다.

4. MapReduce

Map은 Input file을 받아서 key : values 로 반환환다.

Reduce는 통합하여 하나의 결과로 반환한다.

  • MapReduce의 장점은 여러 노드에 걸쳐 high-level의 병렬 작업을 허용하는 기능입니다.
  • Data node는 대량의 데이터를 처리하고 저장하는 데 사용되는 컴퓨터이다.
  • Name node는 HDFS에 있는 데이터를 data node에 분산시키고 관리하는 컴퓨터이다.
  • 다양한 종류의 데이터를 처리할 수 있고 여러 언어를 지원한다.

5. HDFS

하둡의 storage layer를 담당한다.

파일을 블럭들로 나누고 복사본들을 머신들에 저장한다.

하둡과 상호작용 시 커맨드를 사용한다.

핵심 기능으로는

  • Cost efficient : Storage 하드웨어가 저렴하다.
  • Large amount of data : 페타바이트 데이터도 저장 가능하다.
  • Fault tolerant : 한 머신이 고장나도 그 데이터를 다른 곳에서 찾아서 정상화가 가능하다.
  • Scalable : 한 클러스터에 수백개의 노드가 포함될 수 있다.
  • Portable : 다른 플랫폼에서 접근 가능하다.

핵심 개념

  • 블럭

    • 읽히거나 쓰여질 데이터의 최소 단위
    • 64MB 혹은 128MB

  • 노드

    • Primary node 혹은 Name node는 클라이언트의 데이터 접근을 정규화한다.
    • Secondary node 혹은 Data node는 네임노드로 받은 instruction을 수행하게 된다.

  • Rack

    • 같은 네트워크 switch를 사용하는 40~50개의 노드들의 모음이다.

  • Replication

    • 데이터 블럭의 복사본으로 백업을 위한 목적이 있다.
    • Replication factor가 2인 경우 2개의 복사본이 있다.
    • 특정 rack이 고장났을 때를 대비해서 블럭들을 다른 rack에 저장한다.

  • Read and Write concepts

    • write once read many

    • read

      • 클라이언트가 네임 노드에 요청을 보내 블록을 포함하는 데이터 노드의 위치를 제공한다.
      • 클라이언트는 읽기 메소드를 호출하여 모든 파일을 읽고
      • 세션을 종료한다.

    • write

      • 클라이언트가 올바른 권한을 가지고 있는지 확인한다.
      • 이름 노드는 시스템에 파일이 없는지 확인합니다.
      • 파일이 이미 있으면 클라이언트는 IO 예외를 수신합니다.
      • 파일이 없으면 클라이언트는 데이터 노드와 함께 쓰기 권한을 받습니다.
      • 클라이언트가 완료되면, 데이터 노드는 복제본을 만들기 시작하고 클라이언트에 확인을 보냅니다.
